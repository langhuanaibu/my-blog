# CLAUDE.md

本文件是本仓库的项目级最高约束。任何 Agent 进入 `个人博客` 项目后，开始工作前都必须先阅读并遵守本文件。

---

## 1. 项目定位

这是一个个人博客仓库，当前采用以下混合架构：

- 前台：Astro
- 接口：Vercel Serverless Functions（`api/`）
- 数据库：MongoDB
- 后台：保留旧版静态 `admin.html`

这个项目的核心原则不是“推倒重来”，而是：

**在保持线上可用和已有内容稳定的前提下，做渐进式迭代。**

---

## 2. 约束先行

进入本项目后，默认先做这几件事，再开始改代码：

1. 先读本文件 `CLAUDE.md`
2. 再读 `readme.md`，理解当前真实架构、变更记录和踩坑记录
3. 如涉及历史设计背景，再读 `spec.md`、`tasks.md`、`checklist.md`
4. 如涉及项目长期规则，再读 `memory.md`

如果需求和现有规范冲突，**先更新文档约束，再调整代码实现**，不要反过来。

如果当前目录还没有明确规范，不要擅自扩散式开发，不要随意新建杂乱文件。

---

## 3. 工作方式

- 默认使用中文沟通
- 结论先行，再给理由
- 不拍马屁，不做空泛表态，直接给真实判断
- 遇到模糊需求，先给最合理方案；只有在确实影响结果时再追问
- 不为了省事绕过规则，不为了赶进度留下烂尾结构

---

## 4. 修改边界

除非用户明确要求，否则严格遵守以下边界：

- 不改首页核心文案
- 不改已有文章正文内容
- 不改 MongoDB 数据结构
- 不大改 `public/admin.html` 的后台逻辑
- 不擅自重构 `api/` 目录下的业务接口
- 不因为单点需求顺手修改其他页面或模块

如果只是修改某个页面、某个组件或某个功能，**只在对应作用域内动手**。

---

## 5. 目录规则

### 根目录

根目录只允许放项目级文件，不要把临时文件、测试截图、随手导出的报告直接丢在根目录。

根目录当前可接受的内容包括：

- 项目配置文件：如 `package.json`、`astro.config.mjs`、`.nvmrc`
- 项目规范文件：如 `CLAUDE.md`、`readme.md`、`spec.md`、`tasks.md`、`checklist.md`、`memory.md`
- 部署相关文件：如 `CNAME`、`.nojekyll`
- 源码主目录：`src/`、`public/`、`api/`

### `src/`

`src/` 只放 Astro 前台源码。

- `src/pages/`：页面入口
- `src/components/`：可复用前台组件
- `src/layouts/`：页面布局
- `src/styles/`：前台样式

规则：

- 新前台页面优先放在 `src/pages/`
- 可复用 UI 片段优先抽到 `src/components/`
- 不要把后台管理逻辑塞进 `src/`
- 不要把一次性调试脚本直接写进正式页面后长期留存

### `public/`

`public/` 只放静态资源和保留的旧静态页面。

当前已明确包含：

- `public/admin.html`
- `public/script.js`
- `public/style.css`
- `public/images/`

规则：

- 图片放进 `public/images/`
- 可直接通过 URL 访问的静态文件放在 `public/`
- 不要把源码组件、业务逻辑模块塞进 `public/`
- 不要在 `public/` 下制造含义不明的测试文件

### `api/`

`api/` 只放 Vercel Functions。

规则：

- 一个文件对应一个明确接口职责
- 非接口逻辑不要放进 `api/`
- 未经明确要求，不改动已有接口入参与返回结构
- 涉及数据库连接时，优先复用已有连接策略，不要随意重写

---

## 6. 命名规范

- 文件名优先清晰表达用途，不使用 `test_v2`、`new-final`、`临时改改` 这类命名
- Astro 组件使用清晰的 PascalCase 命名
- 普通脚本、接口文件采用已有风格，保持一致
- 新增图片文件名避免无语义命名；如果必须使用上传生成名，确保只出现在资源目录中

---

## 7. 开发原则

### 前台原则

- 用户体验优先于技术炫技
- 优先保持页面清晰、阅读舒适、交互直接
- 不为了“看起来更现代”而破坏原有稳定功能

### 兼容原则

- 这是渐进式迁移项目，新代码必须尊重旧后台和旧接口
- 修改前台时，要考虑 `admin.html`、同域 `/api/*`、历史图片路径是否仍然兼容
- 文章页文件名兼容策略要谨慎，避免破坏旧链接

### 安全原则

- 密钥、Token、密码绝不写进代码
- 依赖环境变量的能力，要明确说明依赖项
- 不为了“先跑起来”而删除鉴权、注释报错或绕开校验

---

## 8. 文档同步规则

只要发生以下情况，必须同步更新 `readme.md`：

- 架构发生变化
- 运行方式发生变化
- Node 版本约束发生变化
- 部署方式发生变化
- 新增值得记录的踩坑经验
- 修复了有复用价值的问题

如果用户要求“创建记忆”，写入 `memory.md`。

如果未来用户要求“创建 skill”，写入 `skill.md`。

---

## 9. 验证规则

只要做了实际修改，原则上都要做对应验证，不允许“只改不验”。

优先使用：

```powershell
npm run build
```

如需本地开发预览，可使用：

```powershell
npm run dev
```

验证时注意：

- 前台页面是否正常构建
- 静态资源路径是否正常
- 旧后台入口 `/admin.html` 是否仍可访问
- 若改动涉及接口联动，明确说明本地是否受环境变量限制

如果没有做某项验证，必须明确说明原因，不要假装已经验证过。

---

## 10. Git 规则

- commit message 使用英文，简洁说明意图
- 不使用破坏性 Git 命令覆盖用户已有改动
- 提交前先确认本次变更作用域清晰

本项目若已完成实际文件修改，默认需要完成版本提交流程；但如果用户明确要求“先别提交/先别推送”，以用户当次指令为准。

---

## 11. 禁止事项

以下行为默认禁止：

- 未读规范就直接开改
- 为了完成任务擅自扩大改动范围
- 在根目录随意堆放临时文件
- 直接删除用户内容来换取“表面成功”
- 用注释报错、跳过校验、硬编码密钥等方式制造伪完成
- 文档和实践长期不一致

---

## 12. 一句话原则

**先定规则，再做改动；先守边界，再谈优化；先验证结果，再宣布完成。**
