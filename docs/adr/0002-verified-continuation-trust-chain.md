# 可信延续独立验证，`day_count` 不作为展示条件

事件登记表由 LLM 匹配产生，`events.json` 中已出现「同类目但不同具体事件被误串」的记录。因此 `day_count ≥ 2` 只说明曾被匹配到同一登记事件，**不代表历史可信**。若直接把 `history_prev` 当事实链喂给生成，会把误串的历史写成读者可见的「来龙」。

决定引入一条独立的信任链：候选匹配之后加**连续性门**（今天这条是否真属于该事件线），通过后再对最近若干条旧历史**逐行验证相关性**，只有验证通过的行才进入生成、对外 `day_count` 与跳转历史。旧登记表不做破坏性修复，运行时只消费验证通过的行。

同时，基于历史生成的 `context/watch` 由一套**独立的轨迹审计**校验，而不是复用通用支撑审计——后者的证据包只含当天原始报道，既看不到历史证据，也无法正确审查这些字段。任一环节失败只让该条退回主 enrich 的已审计结果。

## Consequences

- 展示条件是「可信延续」而非 `day_count ≥ 2`，两者会不一致，这是刻意的。
- 走向的展示覆盖率会低于 100%；**不得为提高覆盖率放宽验证或填充套话**。
- 事件登记表改为先在内存完成全部匹配与生成、最后一次性原子写入，避免留下半成品。
