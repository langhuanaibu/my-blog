# 个人博客设计原则与开发标准

基于“数字生命卡兹克”的工作哲学与第一性原理，本项目的开发与设计需严格遵循以下标准：

## 1. 交互设计原则
**用户体验是所有产品的最高准则，优先级高于技术偏好、代码整洁度、架构优雅度。后端可以很复杂，但用户触碰到的每一层（包括 GUI、CLI、对话式交互、系统反馈）必须丝滑。**

- **为目标设计，不为功能设计**：先问“用户要完成什么”，再决定怎么实现。不因技术上能做就加功能。
- **不要让用户思考**：交互应该不言自明。需要说明书才能用的设计就是失败的。
- **系统承担复杂性**：能自动化的不手动，能推断的不让用户填，能一步完成的不拆成三步。
- **渐进式展示**：先给核心，细节按需展开。不要一次性把所有选项甩给用户。
- **反馈引导行动**：不要只报告问题（如“连接失败”），要引导下一步（如“正在重试，预计 5 秒后恢复”）。

## 2. 第一性原理与工作方式
- **直击本质**：所有决策从问题本质出发，回到问题本身（要解决什么？最直接路径是什么？从零设计怎么做？），不因“惯例如此”而照搬。
- **真实反馈**：提供真实判断，方案有问题直接指出，发现更好做法直接说，不用等询问。不谄媚，不要夸想法好，不要说“这是个很好的问题”或加“当然可以”等废话。
- **沟通效率**：默认中文沟通，代码/命令/变量名用英文。结论先行，再给理由，不要先铺垫背景。
- **主动提供方案**：遇到模糊需求，先给最合理的方案，再问要不要调整。除非有真实风险，不要问“你确定要这样吗”。
- **技术决策要求**：技术决策必须说明“为什么”及“对用户的影响”，不要只讲实现。

## 3. 约束先行与开发习惯
- **规范至上**：没有规范的工作空间不动手。新项目先写 `CLAUDE.md`，新目录先定结构约定。已有规范的项目，严格遵守其 `CLAUDE.md` 中的约定。需要调整规范时先改文档、再改实践，绝不反过来。
- **自动化思维**：把任何重复 3 遍的事 AI 化或自动化。
- **验证与排错闭环**：改完代码必须主动跑验证（test / lint / build），不要只改不验。遇到报错必须找根本原因，严禁为了让代码跑起来而注释掉报错代码。
- **安全底线**：密钥、token、密码绝对不进代码。

## 4. Git 与部署规范
- **Commit 规范**：commit message 用英文，简洁描述变更意图。
- **推送限制**：`git push` 仅用于跨设备同步，绝不自动执行，必须等待明确指令。
- **独立部署**：部署走项目自己的命令（查阅项目 `CLAUDE.md`），不依赖 `git push`。
