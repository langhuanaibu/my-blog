# 日报「来龙 → 现状 → 走向」改造计划

> 状态：设计已定案（2026-07-21 grill-me 访谈及后续复核）。本文件是改动计划，尚未实施。
>
> 排期：先完成 `docs/news_selection_plan.md` 的 7 个有效产出日验收，再实施本计划并启动独立的 5 个有效产出日观察。
>
> 收口：通过验收后，把现行事实并入 `readme.md` 日报章节并删除本文件。

## 1. 问题与已确认事实

用户体感：日报每条新闻「看完就完了」——事件像一个封闭的事实点，看不出它由什么发展而来、接下来取决于什么、什么信号能验证走向。目标是把「过去 / 现在 / 未来」串成一条可信、可继续追踪的脉络。

现有能力与缺口：

- `enrich()` 已生成 `context`（背景）和 `watch`（后续观察），但输入只有当天事件来源和静态兴趣画像，不含跨天历史。
- `track_events()` 在 enrich 与现有支撑审计之后才产生 `event_id / day_count / history_prev`。其中 `history_prev` 只有 `date / title / summary / news_status`，**不保存上一期 `watch`、来源或条目引用**，因此**日报链**现状无法实现「上一期走向是否兑现」的回对。
- 但**周报侧已有成熟范式可复用**：`WEEKLY_*` 已实现 `watch_recap`，含状态枚举 `兑现 | 部分兑现 | 未兑现 | 反转`、`prior` 原文与独立的 `watch_recap_evidence` 证据绑定及审计校验（`daily_news.py:4431-4449 / 4441 / 4446 / 4660`）。日报的走向回对沿用该枚举与证据绑定范式，不另造措辞。
- `build_item`（`daily_news.py:3813`）本就把 `watch` 持久化进每日 data 文件，周报正是据此回读历史 `watch`（`:4731`）。因此取上一期 `watch` 有两条路径：登记表内冗余保存，或凭 `item_ref` 回查前一天 daily 文件。**本计划选前者**——轨迹阶段与登记表一次性原子写入，避免生成期跨文件读取带来的时序与失败面。
- `day_count ≥ 2` 只表示 LLM 曾把两条新闻接到同一登记事件，不代表历史可靠。现有 `events.json` 已出现同类目但不同具体事件被误串的记录，不能把原始 `history_prev` 直接当作可信事实链。
- 当前 `objectivity.mode: interim` 的支撑审计只看当天原始报道。若直接重写基于历史的 `context/watch`，现有审计既看不到历史证据，也无法正确审查这些字段。
- `OBJECTIVITY_FIELD_LIMITS.watch` 目前是 60，但 interim 分支没有统一通过该裁剪路径；只改常量不能保证线上长度一致。已定位到具体位置（`daily_news.py:2290-2304`）：`full_objectivity` 分支逐字段走 `_clip_objectivity_field`，而 else（interim）分支中 `ev["watch"] = r.get("watch", "")` **完全未裁剪**，且 `context` / `significance` 使用**硬编码魔数** `[:80]` / `[:70]` 复制了常量值。
- `.kv.watch` 样式已经存在于 `source/news/news.css`。日报卡片当前没有延续标记；纯文本「延续」只存在于时间线卡片，不能描述为升级日报卡片上的既有标记。

因此，本改造不是简单增加一个二次 prompt，而是建立一条可信轨迹链：

```text
候选匹配 → 连续性验证 → 历史逐行过滤 → 批量轨迹生成
        → 独立轨迹审计 → 原子写入 → 条件式展示
```

## 2. 产品与安全定案

| 决策点 | 定案 |
|---|---|
| 核心体验 | 详情页形成真实的 **来龙 → 现状 → 走向**；过去用于解释现状与前瞻，不是堆背景。 |
| 覆盖粒度 | 所有精选都请求生成「走向」，但只展示审计后仍存在的 `watch`；安全优先，不为 100% 覆盖绕过审计或填充套话。 |
| 走向下限 | 最低包含 **1-2 个关键变量 + 至少一个可观察路标**。历史类比仅在当天来源或已验证历史明确支持时使用，不强制凑三件套。 |
| 表达边界 | `watch` 禁止具体概率数字、无条件断言和来源外类比；独立分析结论进入 `claims[]`，条件式观察路标由轨迹审计直接检查，不在 `claims[]` 重复一遍。 |
| 字段形态 | 不新增读者内容主字段：沿用 `context → 来龙`、`watch → 走向`。允许扩展内部历史元数据和 `history[]` 的嵌套跳转引用。 |
| 可信历史 | `day_count ≥ 2` 不是展示条件。只有「今天与事件主线、最近进展匹配」且至少一条旧历史逐行验证通过，才算可信延续。 |
| 失败策略 | 连续性、生成或审计任一环节失败，都只让该条退回主 enrich 的已审计结果；不阻断日报、不改变精选层级、不显示来龙或延续标记。 |
| 问责方式 | 历史保存最终展示的 `watch`。下一次仅在旧 `watch` 存在且证据足够时做**走向回对**，沿用周报 `watch_recap` 的状态枚举 `兑现 \| 部分兑现 \| 未兑现 \| 反转`；证据不足时不给结论、不猜。不建记分板、不算准确率。 |
| 调用策略 | 连续性验证、轨迹生成、轨迹审计各自按阶段批处理；单条结果按稳定索引校验，缺失或非法时只回退该条。 |

## 3. 管线与数据合同

### 3.1 主 enrich：所有精选的基础「走向」

在 `news-pipeline/daily_news.py` 中调整主 enrich 合同：

1. `ENRICH_SYSTEM` 将 `watch` 定义为「走向」：说明接下来取决于哪 1-2 个变量，并给出至少一个可观察路标。仅在当前来源明确提供既有趋势或可比历史时使用类比。
2. `watch` 统一上限为 **90 个字符**。interim、shadow、active 均调用同一裁剪函数，避免 prompt、常量与运行模式各自限长。具体做法：把 `daily_news.py:2298-2304` 的 else 分支改为与 `full_objectivity` 分支一样逐字段调用 `_clip_objectivity_field`，**一并消除 `context` 的 `[:80]` 与 `significance` 的 `[:70]` 两处硬编码魔数**，使 `OBJECTIVITY_FIELD_LIMITS` 成为唯一长度真源。
3. 主 enrich 仍不接收跨天历史；一次性事件不硬编「来龙」。主支撑审计若删除 `watch`，保留为空，不追加通用占位语或额外修复调用。
4. 当前 enrich 完成并通过既有客观性阶段后，保存每条事件的 `context/watch/claims` 快照，作为轨迹生成或审计失败时的逐字段回退值。

### 3.2 事件匹配与连续性门

保留现有事件候选匹配，但匹配结果不再直接获得可信延续资格：

1. 候选匹配后，新增一次批量连续性验证。每个候选对必须同时满足：
   - 今天的事件与登记事件标题所指向的具体主线一致；
   - 今天的事件与登记表最近一条进展是同一持续事件的后续；
   - 类目一致仍作为代码层硬校验。
2. 候选对验证失败时，不续接旧事件，按新事件登记；不生成来龙、不回对旧走向、不显示延续标记。
3. 候选对通过后，对最近 7 条旧历史逐行验证相关性，只把通过的行放入内存中的 `verified_history_prev`。旧登记表中的无关行暂不迁移或拆分，也不得进入生成、展示天数或跳转历史。
4. 对外 `day_count` 按「验证通过的不同历史日期 + 今天」计算；只有可信计数至少为 2 时才输出 `day_count/history`。
5. 批量响应必须按候选稳定索引返回配对结果和逐行布尔结果。响应结构错误、索引重复、越界、缺项或调用失败时，对受影响候选 fail closed。

### 3.3 轨迹生成：只处理可信延续

对拥有 `verified_history_prev` 的精选事件批量运行轨迹生成：

- 输入仅包括：逐行验证后的最近 7 日历史、历史中可用的最终 `watch` 与来源标识、今天已审计的 `summary/why/context/watch/claims`、今天的原始来源。
- `context → 来龙`：用可信旧进展说明事件怎样走到今天，不补写链外背景。旧 `watch` 存在且材料足以判断时，附一句**走向回对**，状态取自周报同款枚举 `兑现 | 部分兑现 | 未兑现 | 反转`；材料不足则不给状态，也不写「尚待观察」这类占位。回对结论必须由该条的证据包支撑，受轨迹审计检查。
- `watch → 走向`：基于今天事实和可信历史精修关键变量与可观察路标；历史类比可选，不写概率数字或无条件结论。
- 仅允许返回 `idx/context/watch/claims`。字段超长、结构非法或条目缺失时，只回退对应条目的对应字段。

### 3.4 独立轨迹审计

新增与现有通用客观性审计分离的轨迹审计合同，不直接拼接 `SUPPORT_AUDIT_SYSTEM` 与 `OBJECTIVITY_REPAIR_SYSTEM`：

1. 唯一证据包由「逐行验证的历史投影 + 今天原始来源」组成；不得使用未验证登记行、模型常识或联网补证据。
2. 只审查轨迹 pass 新写的 `context/watch/claims`，逐字段返回布尔结果，并逐条校验 claim 的类型与来源归属。
3. 某字段失败时恢复主 enrich 的已审计快照；claim 失败时删除对应 claim。整次审计失败时，该条所有轨迹字段均回退。
4. 轨迹审计不负责降档或改变精选层级。原始主 enrich 结果已经通过现有审计，因此可作为安全回退。
5. 生成和审计都使用批处理；即使整批调用成功，每个条目仍独立校验、独立回退。

### 3.5 登记表版本与原子写入

把事件登记表升级为向后兼容的新版本，旧版直接读取，下一次成功写入时自然升级，不做破坏性迁移。

历史行合同：

```json
{
  "date": "2026-07-21",
  "title": "...",
  "summary": "...",
  "news_status": "发展中",
  "watch": "...",
  "sources": ["来源名"],
  "item_ref": "2026-07-21:pick-123"
}
```

- `watch / sources / item_ref` 均为可选字段；旧历史缺失时不得报错。
- 历史保存的是轨迹审计后的最终 `watch`，不能保存主 enrich 的中间版本。
- 将现有 `track_events()` 的「匹配更新」与「写盘」时序拆开：先在内存完成匹配、连续性验证、轨迹生成和审计，再把今天的最终历史行合入登记表并**一次性原子写入**。
- 同日重跑仍先移除本日旧历史，再按最终结果重建；不得重复增加 `day_count` 或留下同日双记录。
- 新条目的 `item_ref` 使用最终日报 item id。若最终写出前发生异常，登记表保持上一次完整状态，不写半成品。

每日公开数据中的历史投影只输出前端所需字段：

```json
{
  "day_count": 3,
  "history": [
    {"date": "2026-07-20", "summary": "...", "item_ref": "2026-07-20:pick-456"}
  ]
}
```

`history` 只包含逐行验证通过的旧记录；旧行没有 `item_ref` 时仍输出日期与摘要。

## 4. 前端与其他输出

### 4.1 日报卡片

在 `source/news/js/reports.js` 的日报卡片中：

- `item.watch` 存在时，在 `why` 后显示 `<div class="kv watch"><b>走向：</b>…</div>`；不存在时不渲染占位行。
- 只有可信 `day_count ≥ 2` 且存在历史时，新增链接式「第 N 天·延续」标记。它是日报卡片的新入口，不描述为升级卡片上的既有纯文本标记。
- 优先用最近一条历史的 `item_ref` 跳到上一条新闻详情；旧数据缺少 `item_ref` 时退回 `?view=reports&period=day&date=…` 对应日期日报。
- 链接保留 `data-route`，具备键盘焦点和 hover/focus-visible 状态。CSS 复用现有 `.kv.watch`，只补延续链接所需交互样式。

### 4.2 详情页

新闻详情页不再只重排若干 `detail-kv`，而是形成真实的信息顺序：

1. **来龙**：仅在 `context` 存在时显示。
2. **现状**：承载 `summary`、`detail` 与 `why`，避免 summary 继续悬在「来龙」之前破坏顺序。
3. **走向**：仅在 `watch` 存在时显示。
4. **对我的意义**：`significance` 保留为三段之后的独立补充，不混入「现状」。

证据概览、claims、来源和操作按钮继续保留，顺序跟在上述阅读内容之后。

### 4.3 RSS 等输出

- RSS 中现有 `watch` 标签从「后续关注」同步改为「走向」，字段仍为 `watch`，不改 feed 数据语义。
- 其他读取 `watch` 的现有视图继续兼容；只有面向读者的旧标签需要统一，不新增第二套字段。

## 5. 调用、日志与失败隔离

- 近 13 个已有产出日的延续事件为每日 1-8 条。新增链路按阶段批处理，常态新增 3 次 LLM 调用：连续性验证、轨迹生成、轨迹审计；不按事件放大到最多约 24 次。
- 日志至少记录：候选匹配数、连续性通过/拒绝数、被过滤历史行数、轨迹生成成功/回退数、审计字段删除/回退数、最终有走向的精选数。
- 任一新增阶段失败不得阻断主管线。失败事件退回主 enrich；登记表只有在全部内存处理结束后才原子写入。
- 不把「没有生成走向」视为发布失败；把无依据内容进入公开数据视为验收失败。

## 6. 测试与验收

### 6.1 自动化测试

Python 管线测试覆盖：

- 旧版登记表读取、新版可选字段、同日重跑幂等和最终 `watch` 单次写入；
- 同类目但不同事件的候选被连续性门拒绝；
- 正确事件链夹有一条坏历史时，只过滤坏行，不切断其余可信历史；
- `day_count/history` 只按可信历史计算；
- 批量结果缺失、重复、越界、结构非法和调用异常时单条 fail closed；
- 轨迹审计逐字段回退、无依据 claim 删除、整批失败回退；
- interim/shadow/active 的 `watch` 都使用统一 90 字上限，且 `context`/`significance` 的长度只由 `OBJECTIVITY_FIELD_LIMITS` 决定（改常量即改行为，源码内无重复魔数）；
- 走向回对只输出 `兑现|部分兑现|未兑现|反转` 四种状态；旧 `watch` 缺失或证据不足时不产生状态、不写占位语；回对结论无证据支撑时被轨迹审计删除；
- 主支撑审计删掉一次性事件 `watch` 后，不追加占位文案；
- 登记表写入失败或轨迹阶段异常时不留下半成品。

前端测试覆盖：

- 有 `watch` 才显示「走向」，无值不显示；
- 可信延续显示「第 N 天·延续」，一次性或验证失败事件不显示；
- 有 `item_ref` 精准跳上一条详情，无 `item_ref` 按日期降级；
- 详情页 DOM 顺序为来龙 → 现状 → 走向 → 对我的意义；缺失字段时其余顺序稳定；
- RSS 使用「走向」标签；
- 延续链接具备路由属性和键盘可达性。

执行：

```bash
python news-pipeline/tests/test_pipeline.py
npm run test:news
npm run build
```

再用一个包含可信延续、旧坏行和旧版无 `item_ref` 历史的单日夹具做冒烟验证；测试结束清理所有临时文件。

### 6.2 上线验收

必须先完成选材改革的 7 个有效产出日验收，再部署本改造。部署后连续观察 5 个有效产出日：

- 人工审查每天全部可信延续事件；
- 每天再抽查 5 条一次性精选，覆盖不同类目；
- 硬门：错误串线为 0、无依据历史或判断为 0、错误延续跳转为 0、主管线发布失败为 0；
- 质量门：被展示的 `watch` 中，至少 80% 同时具备具体关键变量和可观察路标；
- 单独记录走向展示覆盖率、轨迹回退率和历史行过滤率，不为提高覆盖率放宽安全门。

五日验收失败时关闭轨迹生成与展示路径，保留主 enrich 的 90 字统一裁剪和其他经验证的正确性修复；登记表新增可选字段无需回迁。

## 7. 明确不做

- 不修复、拆分或人工重写全部旧 `events.json`；运行时只消费逐行验证通过的历史。
- 不新增 `context/watch` 之外的读者内容主字段，不建预测记分板、周复盘或准确率仪表盘。
- 不引入 live web 检索、来源外历史类比或具体概率数字。
- 不为了保持卡面整齐而显示通用占位走向。
- 不让轨迹审计改变精选层级，也不扩大现有客观性 active rollout 范围。
- 本计划实施前不扰动选材改革的 7 日观察期。
