# 日报精选选材改革计划（动态阈值 + 保底修复）

> 状态：**已实施，待部署后进入 7 个产出日观察期**。最终事实已同步到 `readme.md`；通过验收并补入实测结果后删除本文件。
>
> 定案日期：2026-07-20（grill-me 访谈，全部决策经用户确认）。本计划**取代** `docs/news_source_roadmap.md` 原 #7 单变量调参。
>
> 时序：**立即实施**，不等 enrich 五日验收结束；对该验收的基线扰动处理见文末。

## 1. 问题与数据证据

用户体感：日报中非 AI 条目偏少。经 2026-07-10 ~ 07-19 十天数据实测（`source/news/data/daily/` 与 `all/`），拆出三个已证实的病因：

### 病因一：pick_max=24 把阈值制变成了固定条数制

10 天里 8 天，被挤到"更多资讯"的 8 条次级条目**全部 ≥ 68 分**（即全部过精选线）。最严重的两天，被挤出条目的分数高于大量入选条目：

| 日期 | 精选类目分布 | 精选最低分 | 被挤出的 8 条次级分数 |
|---|---|---|---|
| 07-10 | ai:15 fin:2 soc:2 wld:2 tech:3 | 75 | 75,75,74,74,74,73,73,72 |
| 07-14 | tech:4 soc:3 ai:3 wld:8 fin:2 | 72 | 74,74,73,72,72,72,72,71 |
| 07-16 | ai:7 fin:2 tech:5 wld:8 soc:1 | 76 | **83,83,82,82,81,81,80,80** |
| 07-17 | ai:12 tech:6 wld:2 fin:2 soc:2 | 70 | **85,84,84,82,81,81,81,80** |
| 07-18 | ai:13 wld:5 tech:2 soc:2 fin:2 | 69 | 74,74,73,73,72,72,72,72 |

### 病因二：构成挤压 + 保底机制两处失效

- AI 大日（07-17/18）AI 占 12-13 席，非 AI 各类被压到保底 2 条。
- **保底会被截断挤掉**：`score_and_select` 先按类补保底、再按分填坑，最后 `sorted(...)[:pick_max]` 一刀截断——保底进的低分条目在热点日又被切掉。实证：07-16 society 精选仅 1 条，低于 `min_per_category: 2`。
- **保底补位无质量下限**：平静日类目不足时从线下补位不设底，07-12 拉入 58 分条目（当日精选最低分 58）。
- 注意：单纯放宽总量治不了构成——07-17 被挤出的 8 条里 6 条还是 AI。

### 病因三：分数通胀，68 分线是"非垃圾线"不是"高质量线"

- `credibility` 全库集中 7-9（sources.yaml），占总分 30% 权重的项对几乎所有条目统一贡献 ~24 分，无区分度。
- 五维打分 prompt（`TRIAGE_SYSTEM`）无校准锚点，LLM 惯性打 7-9：一条普通 T1 新闻底分 ~76。
- 多源加成最高 +8 分，大事件叠满后在 99 分封顶堆积（07-15 一天 4 个 99 分）。
- 结果：热点日 ≥68 分条目（item 级，含同事件多来源重复）达 77-117 条，事件级估 40-60 个。**任何固定分数线在漂移的 LLM 打分分布上都会失效**，这是"纯阈值制、不设上限"不可行的根本原因。

## 2. 定案（均经用户确认）

| 决策点 | 定案 | 理由 |
|---|---|---|
| 病灶 | 总量与构成都治 | 只治总量，AI 大日补进来的还是 AI |
| 阅读预算硬顶 | `pick_max: 24 → 32` | 用户每日通读，+1/3 是可承受上限；超出部分仍有 8 条次级区托底 |
| 类目保底 | `min_per_category: 2 → 3`，并修复"保底被截断"与"补位无下限"两个漏洞 | AI 单类上限 2026-07-15 已撤销，不回头 |
| 分数通胀 | 固定 68 线改为**按产出日等权的动态百分位线**：各历史日先算非纯舆论事件分 nearest-rank P75，再取最近 14 个日值的中位数，钳制 [66,82]，历史不足回退 68 | 避免一次候选暴增的热点日按事件数主导后续阈值 |
| 补位质量下限 | 动态线 − 8 分（当前水位 ≈60），不足则宁缺毋滥 | |
| `pick_min` | 同样遵守动态线 − 8，不足 8 条允许少发 | 防止极端稀日重新拉入低质条目 |
| prompt 锚点 | 首轮不改；7 日验收后仅在动态线频繁触边或头部仍无区分度时单独评审 | 避免算法、账本分布与 prompt 同日变化导致不可归因 |

## 3. 目标状态

### 3.1 config.yaml

```yaml
pick_threshold: 68     # 保留，降级为动态线的回退值
pick_min: 8            # 质量合格时尝试达到，不再无下限硬补
pick_max: 32           # 24 → 32
secondary_count: 8     # 不动
all_view_min_score: 40 # 不动
min_per_category: 3    # 2 → 3
max_per_category: {}   # 维持空

# 新增：动态精选线
pick_dynamic:
  enabled: true
  window_days: 14      # 事件分滚动窗口
  percentile: 75       # 初始 P75，目标过线事件 ~20-26 个/天，第二轮观察后可调
  clamp: [66, 82]      # 防失控钳制区间
  min_history_days: 5  # 账本历史不足 N 天时回退静态 pick_threshold
  backfill_offset: 8   # 类目保留席和 pick_min 共用的补位质量偏移
```

### 3.2 代码改动规格（`news-pipeline/daily_news.py`）

1. **事件分账本**：每日把非 `opinion_only` 事件的最终分写入版本化 `data/score_history.json`；同日重跑覆盖、保留最近 30 个产出日。原子写入失败会 warning，并用静态线重跑当天选材；文件仍位于 `source/news/data/` 自动提交范围，**不扩大 push 例外范围**。
2. **动态线计算**：只读取当天之前的数据；每个历史日先算 nearest-rank P75，再取最近 14 个有效日值的中位数、正数 half-up 取整并钳制到 [66,82]。不足 5 日或账本异常时回退 68；日志与 GitHub step summary 明示阈值来源和历史天数。
3. **`score_and_select` 重构**（现约 1730-1830 行），选材顺序改为：
   - ① 每类保底 3 条作为**保留席**：优先取过线事件，不足时从 ≥（动态线−8）的线下事件补，再不足宁缺；保留席**不参与**最终按分截断；
   - ② 剩余槽位（32 − 保留席数）按分数从过线事件填入；
   - ③ `opinion_only` 封顶从 `pick_threshold − 1` 改为跟随动态线（−1），语义不变；
   - ④ `pick_min: 8` 仅从达到同一补位质量线的事件补齐，不足则少发；
   - ⑤ `max_per_category` 若启用则优先于保留席，且选完后不再做破坏保留席的最终截断。
4. **观测补齐**：`source_health.json` 在原 `count/error` 上增加逐源 `scored_events/selected_events`；Actions 增加选材摘要，shadow 增加高风险单源数量/比率，保留独立链和集中度。
5. **prompt 延后**：首轮保持 `TRIAGE_SYSTEM` 不变，待 7 日选材验收后作为独立变量评审。
6. **测试**：覆盖动态冷启动/损坏/写失败、按日等权 P75、同日幂等、纯舆论排除、热点日保留席、统一质量线、动态 opinion 封顶、逐源观测与摘要脱敏。

## 4. 验收与回滚

- **观察期 7 个有效产出日**（部署后的首个产出日起）：硬门为零质量线以下精选、零纯舆论精选、合格类目保留席零丢失、每日不超过 32 条、动态线不越钳制区间、generate/shadow 无超时或发布失败。
- **趋势门**：相对 2026-07-10~19 基线，“过线但进入次级区”的每日中位数至少下降 50%；四个非 AI 类各自有 3 条合格供给时必须兑现 12 个席位。精选总量只观察，不因稀疏日少于 18 条判失败。
- **回滚**：`pick_dynamic.enabled: false` + `pick_max: 24` + `min_per_category: 2`；保留“席位不被截断”和统一质量下限两个正确性修复，因此这是参数回退而非旧算法完整恢复。账本无须迁移。
- **成本注记**：阶段 B（enrich/detail/客观性审计）调用量随精选条数最多 +33%，deepseek-v4-flash 价位下可接受。
- **对并行验收的扰动**：enrich 安全基线改用部署后前 3 个产出日中位数，五日验收重新计数；客观性 shadow 的 7 日线上门和主管线 14 日信源指标也从观测字段上线后的首个有效产出日重新计数。阮一峰深读源独立于主管线，不重置观察期。
